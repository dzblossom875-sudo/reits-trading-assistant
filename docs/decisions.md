## 交易方向判断方案

- **背景**：交易明细表无明确的"买入"/"卖出"字段，需根据其他字段推断
- **选择**：根据"交割金额"正负判断
  - 交割金额为正 = 卖出（资金流入账户）
  - 交割金额为负 = 买入（资金流出账户）
- **理由**：
  - 与会计记账逻辑一致：正数表示资产增加（卖出获得现金）
  - 与原始数据验证相符（大额正值对应卖出行为）
- **放弃的方案**：
  - 方案B（已注释）：`return "buy" if amount > 0 else "sell"` - 与业务逻辑相反
  - 根据交易类别关键词判断 - 受GBK编码问题影响，不可靠

## Heavy Buy/Sell 阈值计算方式

- **背景**：原逻辑使用 `|净买入|.quantile(0.75)` 同时用于正负方向，导致卖出信号数为0
- **选择**：正净买入和负净卖出分别计算75分位数
  ```python
  buy_threshold = 正净买入的75分位数
  sell_threshold = 负净卖出的75分位数（绝对值）
  ```
- **理由**：
  - 实际交易以净买入为主，负值绝对值普遍较小
  - 对称阈值会导致卖出信号永远不满足
  - 分离计算更符合业务实际（买入激进 vs 卖出保守）
- **放弃的方案**：
  - 对称阈值 `|x|.quantile(0.75)` - 导致卖出信号缺失
  - 固定金额阈值（如1000万）- 无法适应不同规模账户

## Wind API 默认开关

- **背景**：项目支持从Wind API获取行情数据，依赖Wind终端
- **选择**：`USE_WIND_API = True` 默认开启
- **理由**：
  - 实际运行环境已部署Wind终端，可直接获取行情
  - Wind数据更及时准确，板块涨跌幅计算依赖个股行情
  - 已实现本地数据回退机制，Wind失败时自动降级，不会中断流程
- **放弃的方案**：
  - 默认关闭 - 需手动开启才能获取完整板块分析
  - 强制要求Wind - 无回退机制时风险过高

## 分月收益计算口径

- **背景**：需计算各自然月的收益率，存在两种计算方式
- **选择**：本月收益 = (本月末净值 / 上月末净值 - 1)
- **理由**：
  - 与行业惯例一致（月初净值作为起点）
  - 避免首月包含跨月持有收益的问题
  - 便于与人工核对和外部数据源对比
- **放弃的方案**：
  - 月内首个交易日→月末 - 首月会包含跨月持有收益
  - 自然日平均 - 与交易日历不匹配

## 累计净值 vs 单位净值

- **背景**：日报表同时提供"单位净值"和"累计单位净值"
- **选择**：使用"累计单位净值"进行业绩计算
- **理由**：
  - 累计净值包含分红再投资，反映真实收益
  - 单位净值受分红除权影响，会产生断层
  - 与指数收益率计算口径一致
- **放弃的方案**：
  - 单位净值 - 分红后净值下降，导致收益率计算错误
  - 自行计算净值 - 需要处理分红记录，增加复杂度

## 持仓截面 vs 时序

- **背景**：持仓数据用于计算仓位比例，可选择只读最新截面或全时序
- **选择**：同时支持两种读取方式
  - `load_holdings()` - 读最新截面（用于配置偏移分析）
  - `load_holdings_timeseries()` - 读全时序（用于仓位历史图）
- **理由**：
  - 截面数据足够支持配置偏移计算
  - 时序数据需要额外计算，仅在需要仓位图时加载
  - 分离职责，避免单次加载数据量过大
- **放弃的方案**：
  - 只读截面 - 无法生成仓位历史图
  - 只读时序 - 配置偏移分析需要额外处理

## Wind 行情增量缓存策略

- **时间**：2026-03-21 17:57
- **背景**：每次运行都从 Wind API 全量拉取79只个股历史行情，消耗大量 API quota
- **选择**：增量缓存机制，缓存到 `data/processed/wind_prices_cache.csv`
  - 每次读缓存 → 找 max_date → 只拉 `(max_date+1)~today` → 合并去重保存
  - 无缓存时全量拉取（起始日 2024-01-01）
- **理由**：
  - Wind API 有每日调用限额，历史数据不会变化，重复拉取纯属浪费
  - 增量设计对新上市标的友好（新代码在缓存中补 NaN 列，不影响已有数据）
  - 强制刷新只需删除缓存文件，运维简单
- **放弃的方案**：
  - 每次全量拉取：quota 消耗大，日常运行慢
  - 按文件日期判断缓存有效性：精度不足（同一天内多次运行无法区分）
- **关联 Commit**：`bad6117`（2026-03-21 17:57）

## 历史数据与计算数据的对齐方案

- **时间**：2026-03-21 18:09
- **背景**：`history data.xlsx`（2022-11-24起）与日报表计算数据（2026-01-05起）在 BASE_DATE（2025-12-31）对齐，需要拼接成完整时序
- **选择**：缩放系数对齐（rescale 而非截断）
  - 取历史在 BASE_DATE 的值作为 `nav_scale` / `idx_scale`
  - 历史段：直接使用原始值
  - 计算段：`calc_norm × scale` 转换到历史绝对值坐标
- **理由**：
  - 保证拼接处无断层，视觉连续
  - 不丢失计算段的相对精度
  - 验证两段差异为0，说明对齐完全准确
- **放弃的方案**：
  - 截断拼接：拼接处会有跳跃
  - 只用计算段：损失2022-2025年历史
- **关联 Commit**：`df68c7b`（2026-03-21 18:09）

## daily_master.xlsx 列结构设计

- **时间**：2026-03-21 18:23
- **背景**：原有 `daily_tracking.xlsx`（净值+仓位）和 `trade_summary.xlsx`（交易汇总）分散，需要合并为单一主表
- **选择**：以 `full_series`（历史+计算，831天）为底表，左连接 `daily_trades`（仅2026+31天有值）
  - 列顺序与 `history data.xlsx` 同构，便于直接回写更新历史文件
  - 金额全部÷1e4转为万元，与净资产同量纲
  - 历史行（2026前）交易列保留 NaN，不填零（避免误解为"无交易"）
- **理由**：
  - 单一文件降低使用复杂度
  - 与历史文件结构一致，便于日后自动续表
  - NaN vs 0：历史期确实无交易记录，应为 NaN 而非0
- **关联 Commit**：`0f82088`（2026-03-21 18:23）

## X 轴日期格式统一方案

- **时间**：2026-03-21 15:24
- **背景**：各图表 X 轴日期格式不统一（有的只到月，有的到年），手动设置间隔易出错
- **选择**：统一用 `AutoDateLocator` + `ConciseDateFormatter`，封装为 `_apply_date_format(ax)` helper
  ```python
  ax.xaxis.set_major_locator(mdates.AutoDateLocator())
  ax.xaxis.set_major_formatter(mdates.ConciseDateFormatter(ax.xaxis.get_major_locator()))
  ```
- **理由**：
  - 自动根据时间跨度选择合适间隔（日/周/月/年）
  - ConciseDateFormatter 去除冗余信息（年份只在首次出现时显示）
  - 一行封装，所有图表复用，维护成本低
- **关联 Commit**：`aabb4b7`（2026-03-21 15:24）

## 仓位数据来源切换点（2026-03-06）

- **时间**：2026-03-21 19:45
- **背景**：2026-01-01 以后同时有两个仓位数据来源：①history data.xlsx 中记录的历史仓位；②测算值（当日持仓市值/净资产）
- **选择**：≤2026-03-06 使用 history data，≥2026-03-07 使用测算值
- **理由**：
  - history data 中的仓位数据经过人工核对，更可靠
  - 持仓文件非每日记录，测算值在 2026-03-06 之前会因 ffill 出现阶梯形失真
  - 2026-03-07 起持仓文件记录频率提高，测算值更准确
- **实现**：在 `build_full_series()` 中，对 `calc_post`（2026-01-01起）先填入测算值，再用 history_df 覆盖 ≤ 切换点的部分
- **配置方式**：`POSITION_CUTOFF = pd.to_datetime("2026-03-06")` 硬编码在函数内，如需调整直接修改
